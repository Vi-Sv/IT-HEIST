## 📦 ГЛАВА 3: Реверс-инжиниринг (Взлом логики и процессов)## 📑 Чит №8: Инжектор Postman (Ручной впрыск запросов в обход интерфейса)## 🧱 1. Анатомия системы (Суть процесса)
Postman — это специализированный инженерный пульт управления (HTTP-клиент), созданный для тестирования, отладки и документирования API.
Обычный пользователь видит систему только через графический интерфейс (фронтенд) приложения или сайта. Фронтенд — это декорация с жесткими ограничениями (он не дает ввести в поле букву вместо цифры, блокирует кнопки, скрывает технические ID).
Postman позволяет полностью выкинуть интерфейс из цепочки и общаться с бэкендом (сервером) напрямую. Ты можешь руками собрать абсолютно любой HTTP-пакет, вставить любые «грязные» заголовки, подменить тело запроса и симулировать действия любой системы или пользователя.

Запомнить навсегда: Фронтенд сайта — это операционное окно кассы с бронированным стеклом и узкой щелью. Ты можешь просунуть туда только то, что разрешит кассир. Postman — это служебный вход с заднего двора, где ты стоишь напрямую перед кладовщиком (бэкендом) и можешь передать ему в руки коробку любого размера с любым содержимым. Если бэкенд не защищен изнутри, Postman примет даже откровенный мусор.

## 🔀 2. Логическая цепь: Протокол работы аналитика в Postman
Конвейер ручного тестирования и проектирования интеграции состоит из четырех шагов:

   1. Создание рабочего пространства (Workspace & Collection): Запросы не хранятся россыпью. Они пакуются в коллекции (папки проектов, например, Rusagro_Vagon_Project).
   2. Сборка инжектора (Request Construction):
   * Выбираешь метод лома (GET, POST и т.д.).
      * Вбиваешь адрес трубы (URL-эндпоинт, например: https://api.saratov.ru).
      * Заполняешь вкладку Headers (ключи авторизации, тип контента).
      * Заполняешь вкладку Body ➡️ выбираешь режим raw ➡️ тип JSON ➡️ вставляешь текстовое дерево данных.
   3. Впрыск (Send): Нажатие кнопки Send. Запрос улетает по сети, минуя любые браузерные проверки.
   4. Аудит телеметрии (Response Audit): Анализ нижней панели. Смотрим на код ответа (Status), скорость выполнения (Time), размер пакета (Size) и тело ответа (Body JSON/XML).

## 🪤 3. Канализация системных ошибок и капканов (Граничные условия)

* Капкан №1: Ловушка хардкода переменных (Адская рутина).
У тебя есть 20 запросов к тестовому серверу (стенду). Адрес начинается одинаково: https://rusagro.ru.... Перед релизом бэкенд переезжает на прод-сервер: https://rusagro.ru....
* Ошибка: Переписывать адрес руками в 20 запросах. Ты потеряешь час времени и обязательно где-то опечатаешься.
   * Решение Аналитика: Использование Переменных окружения (Environments). В Postman создается переменная {{base_url}}. В строке запроса пишется: {{base_url}}/v1/vagon. При смене стенда ты меняешь значение всего в одной ячейке настроек окружения.
* Капкан №2: Падение авторизации по таймауту (Капкан протухшего токена).
Ты тестируешь методы POST, но сервер раз в 15 минут сбрасывает сессию безопасности и требует новый токен авторизации (Authorization: Bearer <token>).
* Ошибка: Каждые 15 минут запрашивать токен через отдельный запрос, копировать его руками из ответа и вставлять в заголовки всех остальных запросов.
   * Решение: Использование вкладки Authorization на уровне всей коллекции папок. Postman один раз запрашивает токен и автоматически наследует его во все вложенные запросы.
* Капкан №3: Ложное чувство безопасности (Капкан "У меня в Postman всё работает").
Самая классическая ловушка джунов. Ты собрал идеальный запрос в Postman, сервер вернул 200 OK. Ты отдаешь задачу разработчикам фронтенда со словами: «Бэкенд готов, пилите интерфейс». Фронтендеры запускают код из браузера, и запрос намертво блокируется сетевой ошибкой CORS (Cross-Origin Resource Sharing).
* Причина: Postman — это автономная программа, ей плевать на браузерные правила безопасности. А браузер блокирует запросы, если они летят с чужого домена.
   * Решение: Аналитик обязан проверять в ответах Postman наличие заголовков Access-Control-Allow-Origin, разрешающих браузерам забирать эти данные.

## 🎯 4. Срез для собеседования в BigTech (Что спросит Лид)

* Вопрос: Зачем аналитику писать скрипты во вкладках Pre-request Script и Tests в Postman, если это инструмент для ручного тестирования?
* Ответ бьющий в ядро: Для автоматизации конвейера тестирования без кодинга.
* Pre-request Script (выполняется до отправки запроса) нужен, чтобы динамически сгенерировать данные на лету — например, зашить в JSON текущую дату или случайный уникальный ID вагона, чтобы не менять его руками перед каждым кликом Send.
   * Tests (выполняется после получения ответа) нужен, чтобы автоматически проверить результат. Там пишется одна строка кода: pm.response.to.have.status(200);. Если сервер вернет ошибку, Postman загорится красным флагом. Это позволяет запустить прогон всей коллекции из 100 запросов одной кнопкой (Collection Runner) и за 2 секунды проверить работоспособность всего API компании.

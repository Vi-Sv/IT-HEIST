## 📦 ГЛАВА 4: ИТ-Конвейер (Базы, Командный Git и Жизненный цикл)## 📑 Чит №14: Git-flow и управление конфликтами (Захват веток и ручной мерж-контроль)## 🧱 1. Анатомия системы (Суть процесса)
Git — это распределенная система контроля версий. Она хранит историю изменений кода не в виде резервных копий папок, а в виде ориентированного графа коммитов (снимков состояния системы). Каждый коммит знает ID своего родителя.
Git-flow — это жесткий регламент, стандарт управления ветками графа внутри ИТ-команды, защищающий боевой сервер (Прод) от кривого кода.
Анатомия веток в Git-flow делится на два класса:

   1. Постоянные ветки:
   * main (или master) — священная ветка. Здесь лежит только 100% протестированный, рабочий код, который прямо сейчас запущен на проде. Писать код напрямую в main категорически запрещено всем, включая тимлида.
      * develop — предрелизный цех. Сюда сливаются все готовые фичи для комплексного тестирования.
   2. Временные ветки:
   * feature/task-408 — рабочая ветка, которую ты отпочковываешь от develop под свою конкретную задачу. Ты кодишь изолированно от команды. Никто не видит твой мусор, пока ты сам не решишь его сдать.
      * hotfix/ — экстренная ветка хирургического ремонта. Отпочковывается напрямую от main, если на проде обнаружился критический баг (капкан), чинится за час и вливается обратно и в main, и в develop.
   
Запомнить навсегда: Граф Git — это железнодорожная сеть. Ветка main — это главный сквозной путь, по которому без остановок летит скоростной поезд (бизнес компании). Ветка develop — сортировочная станция. Твоя ветка feature — тупиковый ремонтный депо-путь. Ты переводишь стрелку, загоняешь свой вагон в тупик, кодишь, а затем просишь диспетчера (через Pull Request) проверить твою работу и впустить вагон обратно на станцию.

## 🔀 2. Логическая цепь: Жизненный цикл фичи в Git
Конвейер доставки твоего кода на прод состоит из пяти команд:

   1. git checkout develop ➡️ git pull: Переходишь в предрелизную ветку и скачиваешь из центрального репозитория (GitHub) самый актуальный код, который успели накуярить твои коллеги, пока ты спал.
   2. git checkout -b feature/task-408: Создаешь свою изолированную ветку-тупик и автоматически перепрыгиваешь в нее.
   3. git add . ➡️ git commit -m "feat: add vagon validation script": Фиксируешь изменения в локальной истории. Коммит — это гвоздь, забитый в твою ветку графа.
   4. git push origin feature/task-408: Выталкиваешь свою локальную ветку из компьютера на сервер GitHub.
   5. Создание Pull Request (Merge Request): В интерфейсе GitHub ты создаешь запрос на слияние: «Тимлид, я закончил. Перелей мой код из feature/task-408 в общую ветку develop». Тимлид проводит Code Review (аудит кода), пишет замечания и, если всё ок, нажимает кнопку Merge. Фича на станции!

## 🪤 3. Канализация системных ошибок и капканов (Граничные условия)

* Капкан №1: Git Merge Conflict (Адский конфликт слияния).
Самый большой страх джунов. Ты открыл файл script.sql, на строке 42 изменил формулу перерасхода и сделал коммит. Но твой коллега Вася вчера на этой же строке 42 написал свою формулу, и его код уже успели влить в develop. Ты делаешь git merge (или твой Pull Request пытается слиться) — система выдает ошибку: CONFLICT (content): Merge conflict in script.sql. Git останавливает конвейер и замирает.
* Причина: Git — это робот. Он видит, что на одной и той же строке написаны два разных текста. Он не знает бизнес-логики и не может сам решить, чья формула правильная — твоя или Васина. Он блокирует слияние, чтобы вы случайно не затерли код друг друга.
   * Взлом капкана (Ручной мерж): Ты открываешь битый файл руками в редакторе кода. Git услужливо пометил зону конфликта маркерами-замками:
   
   <<<<<<< HEAD
   SELECT volume * 3100 FROM vagon; -- Твой новый код
   =======
   SELECT volume * 1000 FROM vagon; -- Васин код, который уже в develop
   >>>>>>> develop
   
   Ты физически обязан удалить маркеры (<<<<<<<, =======, >>>>>>>), стереть руками неправильную строку (или объединить их, если нужно), сохранить чистый файл, сделать git add script.sql и финальный git commit. Конфликт исчерпан, конвейер поехал дальше.
* Капкан №2: Капкан отставшего поезда (Форсаж без git pull).
Ты кодил в своей ветке две недели. За это время develop ушел вперед на 500 коммитов. Ты пытаешься сделать git push, но сервер отвергает твой пакет.
* Решение Аналитика: Перед отправкой кода ты обязан зайти в свою ветку и выполнить команду git merge develop (или git rebase develop). Ты принудительно вливаешь свежий код станции в свой тупик, локально на своем компьютере находишь и взламываешь все возникшие 50 конфликтов слияния, тестируешь работоспособность гибрида и только после этого отправляешь чистый, адаптированный код на сервер.
* Капкан №3: Заливка секретов на GitHub (.gitignore капкан).
В коде твоего скрипта или таблицы прописаны реальные пароли доступа к базе данных или секретные ключи API. Ты делаешь git push, и твои пароли улетают на публичную витрину GitHub. Через 5 минут роботы-сканеры находят их, и базу твоей компании взламывают.
* Решение: До первого коммита создать в корне проекта файл .gitignore (текстовый список исключений). Записать туда имена всех конфигурационных файлов с паролями. Git будет физически игнорировать их и никогда не пропустит в сеть.

## 🎯 4. Срез для собеседования в BigTech (Что спросит Лид)

* Вопрос: В чем разница между командами git merge и git rebase при слиянии веток, и почему в крупных командах Бигтеха часто запрещают rebase для общих веток?
* Ответ бьющий в ядро:
* git merge берет твою ветку и ветку-цель и создает в графе один новый, финальный коммит слияния (Merge Commit), у которого два родителя. История изменений остается нелинейной, но она честно отражает, кто и когда отпочковался. Это безопасно.
   * git rebase действует как хирургический нож. Он берет все твои коммиты из ветки feature, временно отрезает их, переносит точку отпочкования на самый последний, актуальный коммит ветки develop и насаживает твои коммиты поверх него. История графа становится идеально прямой, как стрела (без петель слияния).
   Но rebase физически переписывает хэши истории коммитов. Если ты сделаешь rebase для общей ветки, в которой параллельно работают другие люди, ты сломаешь им локальные графы версий, у них начнется ад со слиянием, и ты парализуешь работу всей ИТ-команды на полдня. Правило золотого стандарта: rebase — только для своей локальной ветки в тупике, merge — для общих путей.

Замыкаем контур форматов данных: от гибкого JSON переходим к тяжелому, но неубиваемому стандарту XML, на котором до сих пор держится весь банковский финтех, госорганы, сметные выгрузки и ФНС.

##  ГЛАВА 2: Цифровой груз (Форматы данных и Валидация)
## Чит №6: XML и XSD-схемы (Жесткие чертежи и ГЭСН-контроль)
##  1. Анатомия системы (Суть процесса)
XML (eXtensible Markup Language) — это расширяемый язык разметки. В отличие от JSON, он хранит данные не в виде пар ключ-значение, а в виде иерархического дерева тегов. Каждый элемент обязан иметь открывающий тег <tag> и закрывающий тег </tag>. Элементы могут иметь внутренние атрибуты.
Но сам по себе XML — это просто текст. Чтобы ленивый сервер не тратил ресурсы на обработку бракованного XML, используется XSD-схема (XML Schema Definition).

Запомнить навсегда: XML — это заполненный бумажный бланк строгой отчетности (как ГПР или сметная выгрузка). XSD-схема — это прозрачный пластиковый трафарет, который контролирующий орган накладывает сверху на этот бланк. Если хоть одна буква вылезла за границы окошка трафарета, или тип данных в ячейке не совпал с чертежом, XSD-валидатор моментально отбраковывает документ на входе. Сервер даже не начинает его читать.

------------------------------
## 2. Логическая цепь: Сквозная анатомия XML-пакета и XSD-проверки
Бизнес-контекст: переложим задачу учета вагонов Русагро в формат XML:

<?xml version="1.0" encoding="UTF-8"?>   <!-- Техническая стартовая строка -->
<VagonPark station="Саратов-ПТО">        <!-- Корневой тег с атрибутом -->
    <Vagon id="971">                    <!-- Вложенный узел (строка базы) -->
        <Type>Цистерна</Type>           <!-- Текстовое поле -->
        <Volume>3100</Volume>           <!-- Числовое поле -->
        <IsLoaded>true</IsLoaded>       <!-- Логическое поле -->
    </Vagon>
</VagonPark>

Чтобы проверить этот XML, аналитик проектирует XSD-чертеж (схему), где жестко прописывает правила игры:

<xs:element name="Vagon">
    <xs:complexType>
        <xs:sequence>
            <xs:element name="Type" type="xs:string"/>
            <xs:element name="Volume" type="xs:integer"/> <!-- Только число! -->
            <xs:element name="IsLoaded" type="xs:boolean"/>
        </xs:sequence>
        <xs:attribute name="id" type="xs:integer" use="required"/> <!-- Обязателен! -->
    </xs:complexType>
</xs:element>

## 3. Канализация системных ошибок и капканов (Граничные условия)

* Капкан №1: Уязвимость XXE (XML External Entity — Смертельный инжект).
Спецификация XML позволяет подключать к документу внешние ссылки и сущности (<!ENTITY>).
* Ошибка: Если аналитик при проектировании ТЗ забыл отключить поддержку внешних сущностей в парсере сервера, злоумышленник может прислать XML, внутри которого зашит системный вызов. При парсинге сервер послушно выполнит этот вызов, прочитает свои внутренние пароли (например, файл /etc/passwd) и выплюнет их обратно в теле ответа.
   * Решение Аналитика: В требованиях к безопасности интеграции жестко прописывать: «Обработка внешних сущностей (External Entities) в XML-парсере на бэкенде должна быть принудительно отключена (Disabled)».
* Капкан №2: XML-Billion Laughs (Атака "Миллиард ухмылок").
Это аналог «матрешки смерти» из JSON, но реализованный через рекурсивные ссылки внутри тегов.
* Ошибка: Внутри XML создается 10 сущностей, каждая из которых ссылается на предыдущую 10 раз. При чтении маленького файлика размером в 1 Кб парсер начинает разворачивать ссылки по экспоненте. Файл разрастается в RAM сервера до 3 Гигабайт. Процессор загружается на 100%, сервер падает в Out of Memory (OOM).
   * Решение: Жесткое ограничение максимального размера XML-файла на сетевом шлюзе (Nginx) до нескольких мегабайт.
* Капкан №3: Избыточность веса (Капкан оверхеда).
XML весит в 3–5 раз больше аналогичного по смыслу JSON из-за бесконечных дублирующихся строк в закрывающих тегах </tag>. В условиях Highload-трафика Бигтеха это забивает пропускную способность каналов связи и сжигает деньги на сетевую инфраструктуру.

## 4. Срез для собеседования в BigTech (Что спросит Лид)

* Вопрос: В чем разница между понятиями Well-formed XML (Правильно оформленный) и Valid XML (Валидный)?
* Ответ бьющий в ядро:
* Well-formed XML — это документ, который просто соблюдает синтаксис самого языка XML (все теги закрыты, соблюдена вложенность, кавычки на месте). Если в XML-пакете забыли закрыть тег </Vagon>, он перестает быть Well-formed и парсер ломается.
   * Valid XML — это документ, который уже является Well-formed, но дополнительно прошел проверку через внешнюю XSD-схему и полностью соответствует бизнес-требованиям (все типы данных совпали с чертежом, обязательные поля на месте). XML может быть идеально Well-formed, но не Valid, если вместо числа в тег <Volume> прислали текст Три тысячи.
